# Deployment

## Target Architecture

| Component | Platform | Responsibility |
|---|---|---|
| React/Vite frontend | Vercel | Static web application |
| Django backend | Render Web Service | File parsing, regex generation, replacement API |
| OpenAI | External API | Natural-language regex generation |

The MVP does not require a production database. Requests are stateless: the
browser uploads the source file for preview and uploads it again for complete
processing. The backend does not persist files, users, sessions, jobs, or
generated regexes. Render's ephemeral filesystem is therefore acceptable for
this phase. Add PostgreSQL only when persistent application state is introduced.

## Repository Prerequisite

Both platforms deploy from a supported Git provider. Publish the
`regex-pattern-replacement-app` directory as the repository root before
creating the services. The repository root must contain:

```text
render.yaml
backend/
frontend/
docs/
README.md
```

Do not commit `.env`, `OPENAI_API_KEY`, or any copied secret value.

## 2. Deploy the Backend to Render

1. Push the repository to GitHub, GitLab, or Bitbucket.
2. In Render, create a **Blueprint** from the repository.
3. Render reads `render.yaml` and creates
   `regex-pattern-replacement-api`.
4. Enter temporary `CORS_ALLOWED_ORIGINS` and `CSRF_TRUSTED_ORIGINS` values
   such as `https://placeholder.invalid`.
5. Enter `OPENAI_API_KEY` as a secret.
6. Apply the Blueprint and wait for the health check to pass.
7. Record the resulting backend URL:

```text
https://<render-service>.onrender.com
```

The backend API base URL is:

```text
https://<render-service>.onrender.com/api
```

Render automatically supplies `RENDER_EXTERNAL_HOSTNAME`; the Django settings
add that exact host to `ALLOWED_HOSTS`. For a custom backend domain, also set
`DJANGO_ALLOWED_HOSTS` explicitly.

The Blueprint and `backend/.python-version` both pin Python `3.12.13`, avoiding
Render's newer platform default.

### Render Environment Variables

| Variable | Required | Example or source |
|---|---:|---|
| `DJANGO_SECRET_KEY` | Yes | Generated by `render.yaml` |
| `DJANGO_PRODUCTION` | Yes | `true` |
| `DJANGO_DEBUG` | Yes | `false` |
| `CORS_ALLOWED_ORIGINS` | Yes | Final Vercel origin, without trailing slash |
| `CSRF_TRUSTED_ORIGINS` | Yes | Final Vercel origin, without trailing slash |
| `LLM_PROVIDER` | Yes | `openai` |
| `OPENAI_API_KEY` | Yes | Render secret |
| `LLM_MODEL` | Yes | `gpt-5.5` |

## 3. Deploy the Frontend to Vercel

1. Import the same repository into Vercel.
2. Set **Root Directory** to `frontend`.
3. Confirm the detected framework is **Vite**.
4. Add the production environment variable:

```text
VITE_API_BASE_URL=https://<render-service>.onrender.com/api
```

5. Deploy and record the production URL:

```text
https://<vercel-project>.vercel.app
```

`frontend/vercel.json` defines the Vite build, `dist` output directory, and SPA
rewrite. `VITE_*` variables are embedded at build time, so changing the API URL
requires a new frontend deployment.

## 4. Complete Production CORS

Return to the Render service and set both variables to the exact Vercel
production origin:

```text
CORS_ALLOWED_ORIGINS=https://<vercel-project>.vercel.app
CSRF_TRUSTED_ORIGINS=https://<vercel-project>.vercel.app
```

Do not include `/`, `/api`, or another path. Save the environment changes and
restart or redeploy the backend.

Preview deployments use different Vercel hostnames. Keep production restricted
to the exact production origin. If preview deployments must call the backend,
configure a narrowly scoped `CORS_ALLOWED_ORIGIN_REGEXES` value separately.

## 5. Online Verification

Backend health:

```powershell
Invoke-RestMethod "https://<render-service>.onrender.com/api/health"
```

Expected:

```json
{"status":"ok"}
```

Browser workflow:

1. Open the Vercel production URL.
2. Upload `samples/email_sample.csv`.
3. Preview the file.
4. Generate the email regex.
5. Replace matches with `REDACTED`.
6. Confirm the result table and complete-file statistics.
7. Generate an Australian mobile-number regex to verify the real OpenAI
   provider.
8. Confirm the browser console contains no CORS or mixed-content error.

## Operational Notes

- Render free services can have cold-start latency.
- Uploaded rows are processed in memory and are not retained after the request.
- OpenAI calls incur API usage charges.
- Rotate `OPENAI_API_KEY` immediately if it appears in logs, screenshots, Git,
  or chat.
- `DEBUG` must remain false in production.
- HSTS is enabled with a one-hour max age. `includeSubDomains` and `preload`
  remain disabled until a custom-domain policy is deliberately chosen.
- Increase upload or row limits only after testing memory and request-duration
  behavior on the selected Render instance.
